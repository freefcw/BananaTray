# BananaTray UI 弹窗高度自适应折叠布局踩坑与修复文档

本文档详细记录了在开发 BananaTray 弹出窗口时，为了实现 `无界弹窗与内容精准贴合 (Auto-fit Height)` 遇到的一系列极其隐蔽的排查过程及最终完美解法。

## 1. 核心现象与痛点
我们的目标是：弹出框的**初始渲染高度应该正好完整容纳其内部内容**（Header + N张按量卡片 + Footer + 仪表盘）。
而在调试不同数量与数据的 Provider（如 Copilot=1卡, Amp=2卡, Gemini=3卡, Kiro=1卡）时，高度自适应出现了**互相矛盾的薛定谔现象**：
* 按恒定卡片高度 `134px` 计算：Gemini 底部出现了巨大的 30px 空白；但 Copilot/Amp/Kiro 却发生高度短缺，触发极其微弱的内容区滚动，甚至裁剪掉了半个 Dashboard 按钮。
* 按极限测算 `124px` 计算：Gemini 变成刚好滚动，但单卡/双卡的 Copilot 与 Amp 等，底部却多出了大量空白。
* 如果对缺少 `reset_at` 字段的情况删减高度，情况并不改善。

这就导致任何基于线性的单维度尝试拔高或压低 `CARD_HEIGHT` 的估算值，按下了葫芦起了瓢，永远有个特例无法兼容。

---

## 2. 探究根本原因

经过反复测试，结合底层 UI 布局层特性与现象反推，我们锁定了两大幕后真凶：

### 真凶一：GPUI `String::new()` 占位机制与超长文本自动隐式换行
我们一开始以为缺失了 `reset_at` 时间文本的 Provider（如 Copilot, Amp）卡片会更矮，所以尝试在计算窗口时给它们更小的补偿。但实则是：
1. **空文本占位**：在 GPUI 框架下，若以 `child(String::new())` 渲染缺失的文本行，容器模型仍会为其**默认赋予一行 `text_size(11)` 乃至外扩 margin `mt(12)` 的确切高光占比**。所以无论有没有文字，基础物理卡片高度都在 `~130px` 左右不缩减。
2. **灾难性的文本折行膨胀**：那为什么单卡的 Copilot 估高了却在实际反而滚动？因为 Copilot 的标题 `"高级请求 (ENTERPRISE)"` 长度或 Amp 的 `"Claude 3.5 Sonnet (Premium)"` 长度极容易在小窗下**触发自动折叠成两行**。每换行一次，卡片物理高度直接被撑大约 `14px`。我们写死的预估高度由于没算进这 `14px` 的变量，因此窗口总是“包不住”实际被换行拉伸的内容。

### 真凶二：截距（Fixed）与斜率（Card）的方程非线性误差累积
当我们使用强行截断文本折行后，不同 N （张卡片个数）下虽然恢复了线性关系，却依然面临 N=1/N=2 差几像素和 N=3 差几像素**方向相反**（一个缺高度、一个多高度）的终极阻碍。这来源于：
* 弹窗系统/自身封装的无形偏移量（由于 Border、暗调渲染或 Sub-pixel float 舍入带来的细微布局外廓误差）被我们严重低估。
* 卡片本身实际渲染所需的像素斜率，被我们反复通过观察不同特例而高估乃至低估。
这导致计算公式 `W(N) = FIXED + N * CARD` 随着 `N` 增加产生了错位的“剪刀差”。

## 3. 破局之道与最终解决方案

### 步骤1：锁死物理布局的“换行膨胀”变量
为了让 `CARD_HEIGHT` 的函数完全严格线性单调，不允许渲染层私自增加行高，我们在 `src/app/widgets/quota_bar.rs` 给有可能爆长的第一行 Label 和各项数值强制添加了：
```rust
div()
    .flex_shrink_1()
    .overflow_hidden()   // 裁剪溢出
    .whitespace_nowrap() // 禁止其进行跨行折叠
    .child(q.label.to_uppercase()),
```
这彻底杜绝了 Copilot / Amp 等服务因为名字长带来的隐藏行高增加，扫除了变量。

### 步骤2：构建联立方程寻求极值交汇点
我们在三轮极限逼近测试下，分别得出卡片不滚动且不留白的**三阶临界真值测算**：
* 1张卡理想完美高度：~368px
* 2张卡理想完美高度：~505px
* 3张卡理想完美高度：~642px

解三元一次线性关联：
* 斜率（增量步长）：`W(2) - W(1) = 505 - 368 = 137`
* 已知单卡 `CARD_SPACER` 固定占据为 `8px`。因此纯净单卡斜率为：`137 - 8 = 129.0px`。
* 截距补偿（逆向推导）：`W(1) = 195 + 129 * 1 + 44(Dash)` -> 说明我们之前的固定组件相加的 `182`，恰巧丢弃遗漏了 `13px` 的操作系统视窗渲染保留边距或亚像素渲染缝隙！

### 步骤3：更新布局模型标的
将精密解落库至 `src/models/layout.rs`：
```rust
// 修正被遗漏在计算模型外的真实基础架构隐式偏移量
pub const FIXED_HEIGHT: f32 =
    Self::HEADER_HEIGHT + Self::NAV_HEIGHT + Self::CONTENT_PADDING + Self::FOOTER_HEIGHT + 13.0;

// 将卡片高定格为解析所求的唯一解
pub const CARD_HEIGHT: f32 = 129.0;
```

## 4. 总结与启示
在此类 Desktop GUI 编程（无论是 Rust GPUI 乃至 React Native 等）中：
1. 一切使用 “静态计算值” 的动态容器拉伸，其前置条件**必须保证内容不会因为任何内隐机制（换行、Collapse）而自主演化改变高度**。这是静态和动态之间的鸿沟与毒药。
2. 警惕直觉陷阱：当同时发生“增大的情况变小，减小的情况变大”时，**不要头疼医头尝试找个魔法补偿值居间调和**，一定要列等式观察，这百分百只证明你的基数和斜率在两个相悖的方向上同时算错了。
