# GPUI 弹窗高度校准：排查流程与经验教训

> 背景：BananaTray 是一个 macOS 菜单栏 Rust 应用（使用 adabraka-gpui 框架）。弹窗高度需要在 `render()` 中通过 `window.resize()` 显式设置，不支持"内容自适应"。我们需要一个公式 `W(N) = FIXED + N × CARD + (N-1) × SPACER + DASHBOARD` 来精确计算窗口高度。

---

## 1. 问题描述

弹窗底部的"仪表盘"行与 Footer 之间存在**不一致的间隙**：
- 某些 Provider（如 1-2 张卡片的 Copilot/Amp）间隙很大
- 某些 Provider（如 4+ 张卡片的 Antigravity）间隙很小甚至刚好
- 调大常量修好了 A 却弄坏了 B，调小了反过来——典型的"薛定谔现象"

## 2. 错误排查路径（4 轮失败）

### 第 1 轮：从代码推算像素值

**做法**：逐行阅读 UI 代码，手算每个 padding / border / gap / font_size，累加得到常量。

**为什么失败**：
- 假设 text line_height ≈ 1.2~1.3（常见 CSS 经验值），但 **GPUI 默认 line_height = φ = 1.618**（黄金比例）
- 即使知道了 φ，还有一些 GPUI 隐式行为（空字符串占位、亚像素舍入、border 合并）无法从代码中读出
- 最终计算出的常量偏差 10~20px，越改越离谱

**教训**：Desktop GUI 框架的布局引擎有太多隐式行为，纯代码分析无法得到精确像素值。

### 第 2 轮：改变架构——把"仪表盘"移到 scroll 外

**做法**：将"仪表盘"行从 `overflow_y_scroll` 内容区拿出来，放在 scroll 容器和 footer 之间作为固定元素。加上 `min_h_full()` + `flex_1()` 弹性空白。

**为什么失败**：
- 改变了 UI 架构，"仪表盘"始终占据空间且紧贴 footer，用户不接受
- `min_h_full()` 在 GPUI 的 scroll 容器内不生效（Taffy 布局引擎限制）
- 根本没解决常量不准的核心问题

**教训**：不要试图用架构变更来绕过数据不准的问题。先把数据搞对。

### 第 3 轮：按 `reset_at` 有无分别计算卡片高度

**做法**：引入 `CardMetrics { has_reset_at }` 结构体，有 reset_at 的卡片用高常量，没有的用低常量。

**为什么失败**：
- **前提假设就是错的**：GPUI 中 `child(String::new())` 不会让 div 高度变成 0——空字符串仍然分配完整的 text_size 行高空间。所以有无 `reset_at` 的卡片高度是**一样的**
- 两组常量（143/111）都是猜的，且猜错了方向

**教训**：不要基于未验证的假设（"空文本不占空间"）建模。先在运行时确认实际行为。

### 第 4 轮：重新考虑 φ line_height，全部重算

**做法**：发现 GPUI 默认 line_height = φ = 1.618，用这个倍率重算所有文本高度。

**为什么失败**：
- φ 确实是重要发现，但光知道 φ 还不够——还有隐式换行问题没解决
- 某些 label 很长（如 `"Claude 3.5 Sonnet (Premium)"`），在 380px 宽窗口中会**自动折行**，每折一次卡片增高 ~14px
- 这导致卡片高度不是常量，`W(N)` 不是线性函数，公式根本不成立
- 计算出来的值（159px）比实际更离谱

**教训**：即使理论公式正确，如果布局中存在**不可控变量**（隐式换行），公式的前提条件就不成立。

## 3. 正确的解决路径

### 步骤 1：消灭变量——锁死卡片物理高度

在修复常量之前，先确保 `CARD_HEIGHT` 是一个**真正的常量**：

```rust
// quota_bar.rs — 第一行 label
div()
    .flex_shrink_1()
    .overflow_hidden()     // 裁剪溢出
    .whitespace_nowrap()   // 禁止换行
    .child(q.label.to_uppercase()),
```

这一步的意义：
- 消除了 `"Claude 3.5 Sonnet (Premium)"` 等长标签的隐式换行
- 让所有卡片的物理高度严格相同，不因文本内容变化
- 使 `W(N)` 真正变成线性函数，公式前提成立

### 步骤 2：实测三点，解联立方程

放弃代码分析，转为**实际运行测量**：

1. 固定为 1 张卡片，手动微调窗口高度直到"刚好无滚动且无空白" → **W(1) = 368px**
2. 固定为 2 张卡片，同样测量 → **W(2) = 505px**
3. 固定为 3 张卡片，同样测量 → **W(3) = 642px**

由三个实测点解方程：

```
步长 = W(2) - W(1) = 505 - 368 = 137
CARD_HEIGHT = 步长 - CARD_SPACER(8) = 137 - 8 = 129px

截距 = W(1) - CARD_HEIGHT - DASHBOARD_ROW(44) = 368 - 129 - 44 = 195px
验证: W(3) = 195 + 129×3 + 8×2 + 44 = 195 + 387 + 16 + 44 = 642 ✓
```

### 步骤 3：对比理论值与实测值，量化隐式偏移

理论上的固定区域高度（从代码可以算出）：
```
Header(61) + Nav(40) + ContentPadding(22) + Footer(59) = 182px
```

但实测截距 = **195px**，差了 **13px**。

这 13px 就是 GPUI/Taffy 布局引擎的**隐式渲染偏移**——来自亚像素舍入、border 合并等内部行为，从代码分析中**永远无法发现**。

最终常量：
```rust
pub const FIXED_HEIGHT: f32 = HEADER + NAV + CONTENT_PADDING + FOOTER + 13.0;  // 195
pub const CARD_HEIGHT: f32 = 129.0;  // 实测拟合
pub const CARD_SPACER: f32 = 8.0;    // 代码中显式定义
pub const DASHBOARD_ROW_HEIGHT: f32 = 44.0;  // 代码中显式定义
```

## 4. 核心经验教训

### 教训 1：消灭变量是求解常量的前提

> 当"调大 A 好但 B 坏，调小反过来"时，不要试图找一个折中的魔法数字。这个现象百分百说明**模型中存在未控制的变量**（本案例中是隐式换行），导致 W(N) 根本不是线性函数。

正确做法：先找到并消灭变量（`whitespace_nowrap`），让数据回到线性关系，然后再求解常量。

### 教训 2：不要从代码推算像素高度

> GUI 框架的布局引擎存在大量隐式行为（空字符串占位、默认 line_height、亚像素舍入、border box 合并等），这些在源码层面要么找不到，要么需要深入框架内部才能理解。

正确做法：
1. 控制好变量后（消灭换行等不确定因素）
2. 选 3 个不同 N 值做实测，得到 3 个精确的 `(N, W)` 数据点
3. 用联立方程求解 CARD_HEIGHT 和 FIXED_HEIGHT
4. 交叉验证（用第 3 个点检验前两个点算出的公式）

### 教训 3：GPUI 特有的隐式行为备忘

| 行为 | 影响 | 应对 |
|------|------|------|
| 默认 line_height = φ = 1.618 | 未设 line_height 的文本高度 = font_size × 1.618 | 关键文本显式设 `.line_height(relative(1.0))` |
| `child(String::new())` 仍分配行高 | 空文本 div 不会高度坍缩 | 用 `.children(option.map(\|...\|))` 条件渲染 |
| 未设 `whitespace_nowrap` 时自动折行 | 长标签在窄窗口中隐式增加行高 | 对固定高度区域的文本一律加 `.overflow_hidden().whitespace_nowrap()` |
| 约 13px 的系统级隐式偏移 | 理论计算值与实际值存在固定偏差 | 只能通过实测发现，无法从代码推导 |

### 教训 4：解决布局问题的正确工作流

```
1. 观察现象 → 是否存在"薛定谔现象"（调大调小都不对）？
   ├── 是 → 说明模型中有未控制的变量，先找到并消灭它
   └── 否 → 单方向偏差，可能只是常量不准

2. 消灭变量 → 加 whitespace_nowrap / 条件渲染 / 固定尺寸

3. 实测校准 → 选 3 个数据点实测，解联立方程求常量

4. 交叉验证 → 用第 3 个点验证公式
```
